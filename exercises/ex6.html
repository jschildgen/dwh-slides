Exercise 1: Data Integration

This exercise is the continuation of the previous exercise sheet. Make sure that you have a staging area with tables that contain data from different sources (Sheet 5, Exercises 1.3 and 2) and empty tables in your integrated schema (Sheet 5, Exercise 1.6).

The following exercises serve only as a rough guide. Depending on your scenario, your source tables, target tables, and your data, it can be that several tasks do not make sense. Finally, it's now your job to tell yourself what to do to integrate your data in your data warehouse. Document what you do and why.

In the next workbook (Challenge 4), there will only be a short question about this exercise sheet. You need to write a few words about how you performed your data integration and data cleaning.

  1. In which of your source tables do you have errors or duplicates in your data? For those, add a column error_info of type VARCHAR(2000).
  2. For each of your individual source tables (not cross-table!), find out whether there are duplicate rows (rows describing the same real-world entity). For the matching rows, set the error code to an appropriate error message.
  3. Which other errors are there? Set appropriate error messages to the erroneous rows.
  4. Now perform entity-matching across multiple source tables. For each row in one source table, find matching rows in the other table, using (a) equal join, (b) similarity join with a Levenshtein less than an appropriate threshold, (c) similarity join using Soundex. For each approach, write down the number of rows that find a match.
  5. Which entities find a match with Levenshtein that do not find a match with Soundex and vice versa? Are there false positives (match found but it shouldn't be a match)? Are there false negatives (it should be a match but no match was found)?
  6. Choose an appropriate matching method (equal join, Levenshtein, or Soundex) to integrate the data from the source tables into the target table.
  7. Also integrate the rows that only appear in one of the source tables in the target table.
  8. Integrate all other data also in your DWH schema. Skip the erroneous rows (or fix the errors somehow). Perform entity matching, data integration, deduplication, data scrubbing, data cleaning, and normalization.

Remark: Most tasks on this exercise sheet are made for String values in your data. These values are typically in your dimension tables. Examples: city names, person names, song titles, product names, categories, ... For numeric data you can only check for errors (are the values in a meaningful range, outlier detection, ...). For dates and timestamps (and strings/numbers where suitable), convert them into a homogeneous form.
